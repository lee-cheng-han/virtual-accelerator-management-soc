# Minimal RISC-V Management Subsystem

## Status and scope

The repository contains a bootable RV32 management subsystem consisting of an
out-of-tree QEMU patch series, freestanding bring-up firmware, and a Zephyr
control plane. It is not yet the PCIe accelerator model.

Implemented:

- one RV32 hart with the `rv32` QEMU CPU model;
- immutable reset ROM generated by QEMU;
- 512 KiB SRAM containing the firmware image, stack, and data;
- a memory-mapped 16550-compatible UART;
- ACLINT machine software and timer interrupt blocks;
- startup assembly, linker layout, trap entry, UART output, SRAM test, and a
  machine-timer interrupt test;
- a smoke test that verifies the complete boot transcript.

The QEMU machine also implements the firmware-facing command portal, mailbox,
watchdog, reset, and telemetry regions. Those behaviors and the Zephyr drivers are documented
in the [management-peripheral contract](management-peripherals.md) and
[Zephyr board guide](zephyr-board-port.md). PCIe, BAR registers, DMA,
accelerator engines, the Linux driver, userspace, and PCI queue integration
remain outside this subsystem. The portal deliberately uses the production
descriptor/completion ABI while remaining private to the firmware harness.

## Platform contract

| Region | Base | Size | Purpose |
|---|---:|---:|---|
| Reset ROM | `0x00001000` | 64 KiB | QEMU-generated reset vector |
| ACLINT software interrupt | `0x02000000` | 16 KiB | Machine software interrupt |
| ACLINT machine timer | `0x02004000` | 32 KiB | `mtimecmp` and `mtime` |
| UART | `0x10000000` | 8 bytes | 16550 register window |
| Mailbox | `0x10010000` | 4 KiB | Firmware event bridge |
| Management control | `0x10020000` | 4 KiB | Watchdog, reset, telemetry |
| Command portal | `0x10030000` | 4 KiB | Descriptor/completion staging |
| SRAM | `0x80000000` | 512 KiB | Firmware, data, and stack |

The reset ROM transfers control to the ELF entry point in SRAM. The firmware
initializes `gp` and `sp`, clears `.bss`, installs its machine trap vector, and
prints the first three boot checkpoints. It then programs `mtimecmp`, enables
machine timer interrupts, and waits in `wfi`. The trap handler advances the
timer comparison and records the interrupt; only then does main print
`UART: ready`. The last checkpoint therefore proves that instruction fetch,
SRAM, UART, CSR access, ACLINT timer delivery, trap entry, and return all work.

The UART interrupt output is directly connected to the hart's machine-external
interrupt input. A PLIC is intentionally absent from this minimal topology.

## Build the firmware

The firmware needs a bare-metal RISC-V GCC toolchain that provides GCC,
binutils, and `objcopy`. The default prefix is `riscv64-unknown-elf-` even
though the generated image is RV32. The recorded validation used the Xilinx
2025.2 RISC-V toolchain with GCC 13.4.0.

```sh
make firmware CROSS_COMPILE=riscv64-unknown-elf-
```

The ELF and raw binary are written under `build/firmware/baremetal/`.

## Integrate the QEMU machine

The patch was compiled and tested against QEMU commit
`94826ec1370328375c3b6d1e80fdc94c8f46c348` (QEMU 11.0.50 development tree).
First verify that it applies to the selected clean QEMU source tree:

```sh
make qemu-patch-check QEMU_SRC=/path/to/qemu
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0001-hw-riscv-add-vams-riscv-machine.patch"
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0002-hw-add-vams-management-peripherals.patch"
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0003-hw-misc-add-vams-pcie-endpoint.patch"
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0004-hw-misc-add-vams-nop-queue-transport.patch"
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0005-hw-misc-add-vams-firmware-command-portal.patch"
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0006-hw-misc-bridge-vams-pci-queues-to-firmware.patch"
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0007-hw-misc-add-vams-memory-copy-engine.patch"
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0008-hw-misc-add-vams-memory-fill-engine.patch"
git -C /path/to/qemu apply \
  "$PWD/qemu/patches/0009-hw-misc-add-vams-crc32-engine.patch"
```

Configure that tree with `riscv32-softmmu` and `x86_64-softmmu` to build both
the management machine and PCIe endpoint validation targets. Build
`qemu-system-riscv32` and `qemu-system-x86_64` using the normal QEMU build
procedure. The series adds the machine name `vams_riscv` and device type
`vams-pcie`; it does not modify an installed system QEMU.

## Validate the boot

Pass the built QEMU executable and cross-toolchain prefix to the smoke target:

```sh
make smoke \
  CROSS_COMPILE=riscv64-unknown-elf- \
  QEMU_SYSTEM_RISCV32=/path/to/qemu/build/qemu-system-riscv32
```

Expected result:

```text
VAMS RISC-V boot smoke test: PASS
```

For an interactive boot, run:

```sh
/path/to/qemu-system-riscv32 \
  -M vams_riscv -display none -monitor none -serial stdio \
  -bios build/firmware/baremetal/vams-riscv-fw.elf
```

Terminate QEMU with `Ctrl-C` after the four boot messages. The firmware enters
an idle `wfi` loop after completing bring-up.

## Deliberate limitations

- The topology is fixed at one hart and exactly 512 KiB of SRAM.
- ELF firmware must be supplied explicitly with `-bios`; ELF-backed loading
  restores initialized SRAM contents on management reset.
- There is no device tree because the freestanding firmware uses a fixed,
  shared platform header.
- The smoke test uses GNU `timeout` and expects QEMU to remain in the idle loop.
- The patch is maintained out of tree and must be rebased when its QEMU APIs
  change.
